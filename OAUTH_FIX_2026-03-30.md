# OAuth Authentication Fix - March 30, 2026

## Issue
Google OAuth callback was failing with 400 Bad Request error:
```
GET http://sfpl.mysalesforcedemo.com/api/auth/google/callback?state=...&code=... 400 (Bad Request)
```

## Root Cause
Cookie security configuration mismatch:
- `COOKIE_SECURE=true` was set (requires HTTPS)
- All application URLs were configured with `http://` instead of `https://`
- When OAuth flow redirected back to HTTP callback, browsers refused to send the secure session cookie
- Without the session cookie, state/nonce validation failed → 400 error

## Solution Applied

### 1. Updated Heroku Config (v10)
Changed all URLs from HTTP to HTTPS:
```bash
heroku config:set \
  APP_BASE_URL=https://sfpl.mysalesforcedemo.com \
  CORS_ORIGIN=https://sfpl.mysalesforcedemo.com \
  GOOGLE_CALLBACK_URL=https://sfpl.mysalesforcedemo.com/api/auth/google/callback \
  --app aosfpl
```

### 2. Updated Google OAuth Console
Updated authorized redirect URI in Google Cloud Console:
- OLD: `http://sfpl.mysalesforcedemo.com/api/auth/google/callback`
- NEW: `https://sfpl.mysalesforcedemo.com/api/auth/google/callback`

### 3. Restarted Application
```bash
heroku restart --app aosfpl
```

## Verification
- Health check: `https://sfpl.mysalesforcedemo.com/api/health` → `{"ok":true}`
- Login page accessible: `https://sfpl.mysalesforcedemo.com/login`
- OAuth flow now works correctly with HTTPS callback

## Key Learnings
1. When `COOKIE_SECURE=true`, **all** application URLs must use HTTPS
2. Session cookies won't be sent by browsers on HTTP requests when marked secure
3. OAuth state validation requires the session cookie to be present
4. Always verify protocol consistency across: app URLs, callback URLs, and cookie settings
